## 二、项目打包架构（先摸清现状）

这个项目**已经配置好了完整的 Tauri v2 桌面打包基建**，不需要从零搭：

- **打包框架**：Tauri v2（`client/src-tauri/`），Rust 后端外壳 + WebView 前端
- **打包目标**：Windows NSIS 安装包（`.exe`）
- **后端方案**：Node/Express 服务器作为 **sidecar 二进制** 随包分发，桌面端启动时自己拉起后端，绑定 `127.0.0.1:8787`
- **打包资源**：`server-dist`（后端生产 bundle）+ 项目初始化模板（`.claude/.codex/.agents/AGENTS.md`）
- **自动更新**：配置了 updater，指向 GitHub Releases 发布仓 `https://github.com/LYGOLANG/fufan-cc-flow-releases/releases/latest/download/latest.json`（2026-07 从自有服务器切换而来，源码仓私有、发布仓公开）

### 关键文件与路径

| 用途 | 路径 |
|------|------|
| 打包编排脚本 | `scripts/package-desktop.mjs` |
| sidecar 准备脚本 | `client/scripts/prepare-sidecar.mjs` |
| Tauri 配置 | `client/src-tauri/tauri.conf.json` |
| Rust 主逻辑 | `client/src-tauri/src/sidecar.rs`（`SERVER_PORT = 8787`） |
| Rust 依赖清单 | `client/src-tauri/Cargo.toml`（依赖本地 crate `desktop/crates/cc-core`） |
| 打包产物输出 | `release/` |
| 后端 bundle（打进包） | `client/src-tauri/server-dist/` |
| sidecar node 二进制 | `client/src-tauri/binaries/node-<target-triple>.exe`（打包时生成） |

### 打包流水线原理（`pnpm package:desktop` 内部）

```
pnpm package:desktop
  └─ node scripts/package-desktop.mjs
       └─ pnpm --filter client tauri build
            ├─ beforeBuildCommand: pnpm prepare:sidecar && pnpm build
            │    ├─ prepare:sidecar（client/scripts/prepare-sidecar.mjs）
            │    │    ├─ 构建 server（pnpm --filter server build）
            │    │    ├─ 组装生产 bundle 到 server-dist/（含 dist + package.json）
            │    │    ├─ pnpm install --prod --node-linker=hoisted（扁平 node_modules，避免符号链接）
            │    │    ├─ 拷贝项目模板（.claude/.codex/.agents/AGENTS.md）进 server-dist/
            │    │    └─ 复制当前 node runtime 作为 sidecar 二进制
            │    └─ pnpm build（构建前端 SPA + server）
            ├─ cargo 编译 Rust（首次最慢）
            └─ NSIS 打包成安装器
       └─ 把安装器从 target 拷贝到 release/
```

> **关键设计**：`prepare-sidecar.mjs` 特意用 `pnpm install --node-linker=hoisted` 生成**扁平的 node_modules**，而非 pnpm 默认的符号链接结构。因为打包后的 sidecar 独立运行时，Node 解析器需要真实的扁平依赖，符号链接会导致 `ERR_MODULE_NOT_FOUND`（且 release 构建没有日志 sink，报错是静默的）。

## 四、过程中踩的两个坑（重要）

### 坑 1 · 打包中途夭折（中断导致编排脚本被杀）

**现象**：首次打包时，两次手动中断后，`cargo` / `rustc` 进程仍在跑（CPU 累积），但最终没有产出安装包，日志文件 0 字节。

**根因**：打包命令用了 `pnpm package:desktop | tail -80`，中断的 SIGINT 杀掉了带 `tail` 的 shell 头，`cargo` 被**孤儿化**继续编译，但**编排脚本 `package-desktop.mjs` 死了**——所以 cargo 编译完成后，负责 NSIS 打包和拷贝产物的那一步没人执行，最终无安装包；`tail` 被杀也导致日志全丢。

**解决**：
1. 打包命令**不要用 `| tail`**，直接 `> build-desktop.log 2>&1` 全量落盘，中途可读进度
2. 用 `run_in_background` 后台跑，编排脚本不受终端中断影响
3. Rust 编译缓存复用，第二轮很快干净出包

### 坑 2 · 静默安装装不上（NSIS `/S` 在非交互会话不落文件）

**现象**：`Start-Process "setup.exe" -ArgumentList "/S"` 后，安装器约 5 秒退出，注册表写了卸载项，但 `%LOCALAPPDATA%\Agent Flow` 目录**完全为空**（0 文件），连 `uninstall.exe` 都没有。

**根因**：NSIS 安装器静默安装时需要 UAC 提权，非交互会话没人批准 → 提权失败 → 只写了注册表项、解压文件阶段失败。这是**环境限制，不是安装包损坏**。

**解决**：
1. 清掉注册表残留（避免污染系统）：
   ```powershell
   Get-ChildItem "HKCU:\...\Uninstall","HKLM:\...\Uninstall" |
     ForEach-Object { $p = Get-ItemProperty $_.PSPath; if ($p.DisplayName -match 'Agent Flow') { Remove-Item $_.PSPath -Recurse -Force } }
   ```
2. 改用 **7-Zip 解压安装器** 得到等价布局做冒烟（见步骤 4）
3. 真实安装：用户**双击**安装器、点 UAC 同意即可正常装到 `%LOCALAPPDATA%\Agent Flow`

## 五、环境相关注意事项（本机特有）

1. **本机全局代理**：设了 `HTTP_PROXY/HTTPS_PROXY=127.0.0.1:7897`，`curl`/`Invoke-WebRequest` 打 localhost 会被代理劫持。测本地后端必须绕过代理：`Invoke-WebRequest -Proxy $null` 或 `curl --noproxy '*'`。
2. **终端输出乱码**：本会话终端有回显串扰（输出重复、夹杂乱码），关键结果均通过「写文件 + Read 读回」的方式取得可靠数据。
3. **node-pty 需本机编译**：`pnpm install` 会触发 node-gyp 编译，需 Python + C++ 构建工具就位。

```bash

  --title "v<版本>" --notes "更新说明" \
  release/updates/AgentFlow_<版本>_x64-setup.exe release/updates/latest.json
```
